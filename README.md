# SQL Data Lineage

A comprehensive tool for extracting, analyzing, and visualizing SQL data lineage. This project parses SQL scripts to build a complete map of table-to-table and column-to-column data dependencies, enabling data engineers and analysts to understand data flow across complex SQL pipelines.

## Features

- **SQL Parsing & Lineage Extraction**: Parses complex SQL queries including CTEs, JOINs, UNIONs, and subqueries
- **Table-Level Lineage**: Identifies all source tables and target tables in SQL statements
- **Column-Level Lineage**: Tracks data flow at the column level from source to target
- **Bulk Processing**: Scan directories for multiple SQL files and extract lineage from all of them
- **Interactive Visualization**: Web-based UI to explore lineage graphs with zoom, pan, and filtering
- **Graph Search**: Find upstream dependencies for any table with DFS traversal
- **Multiple SQL Dialects**: Leverages sqlglot for parsing common SQL dialects
- **JSON Export**: Output lineage data in structured JSON format for integration with other tools

## Quick Start

### 1. Extract Lineage from SQL Files

```bash
python main.py scan ./sql_scripts --output lineage.json
```

This command:
- Scans the `sql_scripts` folder for all `.sql` files
- Extracts lineage from each file
- Outputs results to `lineage.json`

### 2. Visualize the Lineage Graph

```bash
python main.py viz lineage.json
```

Opens an interactive web UI at `http://localhost:8001` where you can:
- View the complete lineage graph
- Search for specific tables
- Click on tables to see column-level lineage
- Explore upstream dependencies

## Usage

### Command Line Interface

#### Scan Command

Scan a folder for SQL files and extract lineage:

```bash
python main.py scan <input_folder> --output <output_file> [--target <table_name>]
```

**Arguments**:
- `input_folder` (required): Path to folder containing SQL scripts
- `--output` (required): Path to output JSON file
- `--target` (optional): Filter results to extract lineage only for a specific target table

**Examples**:

Extract lineage from all SQL files:
```bash
python main.py scan ./sql_scripts --output lineage.json
```

Extract lineage only for the `sales_summary` table:
```bash
python main.py scan ./sql_scripts --output lineage.json --target sales_summary
```

#### Viz Command

Visualize the lineage from a JSON file:

```bash
python main.py viz <lineage_json>
```

**Arguments**:
- `lineage_json` (required): Path to lineage JSON file generated by the scan command

**Example**:
```bash
python main.py viz lineage.json
```

### Web Visualization

Once you run the `viz` command, the application starts a FastAPI server at `http://localhost:8001`.

#### Features:

- **Graph View**: Interactive DAG (Directed Acyclic Graph) showing all table dependencies
- **Search**: Type a table name and press Enter to filter the graph to show only upstream dependencies
- **Column Lineage**: Click on any table node to see column-level lineage in the right panel
- **Node Styling**: 
  - Blue nodes: Tables in the lineage
  - Red nodes: Currently selected table
  - Arrows: Show data flow direction

## How It Works

### 1. SQL Parsing (extractor.py)

The extractor uses **sqlglot** to parse SQL statements and extract:

- **Target Tables**: Identified from INSERT, CREATE, UPDATE, DELETE, MERGE statements
- **Source Tables**: Extracted from FROM and JOIN clauses
- **CTEs (Common Table Expressions)**: Tracked and normalized
- **Column Lineage**: Maps output columns to their source columns through the SQL transformations

**Process**:
1. Parse SQL using sqlglot parser
2. Find target table (destination)
3. Collect all source tables from the query
4. Filter out CTEs from the source tables (they are internal transformations)
5. Trace column mappings through SELECT projections

### 2. File Scanning (scanner.py)

- Recursively walks through directories
- Identifies files with `.sql` extension
- Reads file contents for processing

### 3. Graph Building (graph_builder.py)

- Converts lineage records into a graph structure
- Creates nodes for each table
- Creates edges representing table-to-table dependencies
- Output format suitable for visualization libraries like Cytoscape

### 4. Column Lineage Mapping (column_builder.py)

- Builds a dictionary mapping each target table to its columns
- For each column, tracks the source columns it derives from
- Supports multiple source columns (e.g., in expressions like `revenue = amount + tax`)

### 5. Search & Filtering (search.py)

- Implements Depth-First Search (DFS) to find all upstream dependencies
- Given a table name, traverses the graph backwards to find all source tables
- Returns filtered graph containing only relevant nodes and edges

### 6. Web Visualization (viz.py)

- FastAPI application serving the lineage data
- Provides REST APIs for graph retrieval and searching
- Serves static frontend files (HTML/CSS/JS)
- Uses Cytoscape library for interactive graph rendering

## Examples

### Example 1: Simple INSERT Statement

**SQL Input**:
```sql
CREATE TABLE sales_summary
SELECT 
    customer_id,
    SUM(amount) AS revenue,
    COUNT(*) AS total_orders
FROM orders
GROUP BY customer_id
```

**Output**:
```json
{
  "target_table": "sales_summary",
  "source_tables": ["orders"],
  "cte_sources": [],
  "column_lineage": [
    {
      "target_column": "customer_id",
      "source_columns": ["orders.customer_id"]
    },
    {
      "target_column": "revenue",
      "source_columns": ["orders.amount"]
    },
    {
      "target_column": "total_orders",
      "source_columns": []
    }
  ]
}
```

### Example 2: Complex Query with CTEs and JOINs

**SQL Input**:
```sql
WITH customer_orders AS (
  SELECT 
    o.customer_id,
    o.amount,
    o.order_date
  FROM orders o
  JOIN customers c ON o.customer_id = c.id
),
summary AS (
  SELECT 
    customer_id,
    SUM(amount) AS total_amount,
    COUNT(*) AS order_count
  FROM customer_orders
  GROUP BY customer_id
)
CREATE TABLE customer_summary
SELECT 
  customer_id,
  total_amount,
  order_count
FROM summary
```

**Output**:
```json
{
  "target_table": "customer_summary",
  "source_tables": ["customers", "orders"],
  "cte_sources": ["customer_orders", "summary"],
  "column_lineage": [
    {
      "target_column": "customer_id",
      "source_columns": ["customer_orders.customer_id"]
    },
    {
      "target_column": "total_amount",
      "source_columns": ["customer_orders.amount"]
    },
    {
      "target_column": "order_count",
      "source_columns": []
    }
  ]
}
```

## API Endpoints

The visualization server provides the following REST endpoints:

### GET /
Returns the main HTML interface

### GET /graph
Returns the complete lineage graph structure
```json
{
  "nodes": [
    {"id": "orders"},
    {"id": "customers"},
    {"id": "sales_summary"}
  ],
  "edges": [
    {"source": "orders", "target": "sales_summary"},
    {"source": "customers", "target": "sales_summary"}
  ]
}
```

### GET /lineage/{table}
Returns upstream lineage for a specific table (DFS traversal)

**Parameters**:
- `table` (URL parameter): Table name to search for

**Response**: Graph structure containing only the upstream dependencies

### GET /column-lineage/{table}
Returns column-level lineage for a specific table

**Parameters**:
- `table` (URL parameter): Table name to retrieve column lineage for

**Response**: Dictionary mapping target columns to source columns
```json
{
  "customer_id": ["orders.customer_id"],
  "revenue": ["orders.amount"],
  "total_quantity": ["orders.quantity"]
}
```

## Output JSON Format

The lineage JSON file contains an array of objects, each representing a SQL file:

```json
[
  {
    "file": "/path/to/sql_file.sql",
    "lineage": [
      {
        "target_table": "table_name",
        "source_tables": ["source1", "source2"],
        "cte_sources": ["cte_name"],
        "column_lineage": [
          {
            "target_column": "col_name",
            "source_columns": ["source_col1", "source_col2"]
          }
        ]
      }
    ]
  },
  {
    "file": "/path/to/another_file.sql",
    "error": "Error message if parsing failed"
  }
]
```

## Error Handling

If a SQL file cannot be parsed, the error is captured and included in the output:

```json
{
  "file": "/path/to/problematic_file.sql",
  "error": "Error message describing the parsing issue"
}
```

This allows batch processing to continue even if individual files have issues.

## Limitations

- **SQL Dialect Support**: Relies on sqlglot's parsing capabilities; some advanced or dialect-specific SQL may not be fully supported
- **Column Lineage**: Complex expressions and window functions may have limited column tracking
- **Performance**: Very large SQL files or deeply nested queries may require more processing time
- **UDF Functions**: User-defined functions are not resolved to their implementations

## Tips & Best Practices

1. **Organize SQL Files**: Group related SQL files in subdirectories for easier management
2. **Naming Conventions**: Use consistent table and column naming for better lineage clarity
3. **Version Output**: Keep dated versions of lineage outputs for change tracking
4. **API Integration**: Use the JSON output to integrate with data cataloging tools
5. **Regular Scans**: Re-scan directories periodically to keep lineage data current
